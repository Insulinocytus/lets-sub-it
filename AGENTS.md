# AGENTS.md

## Project Overview

Lets Sub It is a self-hosted YouTube subtitle generation and translation project. It has three main modules:

- `backend/`: Go 1.22 HTTP API for job submission, job tracking, SQLite persistence, downloading/transcribing media, and LLM translation orchestration.
- `whisper/`: Python package `whisper-cli` for local Faster Whisper transcription and validated VTT output.
- `extension/`: Chrome Manifest V3 extension built with WXT, Vue 3, TypeScript, Tailwind CSS, and shadcn-vue components. It submits YouTube subtitle jobs and renders completed subtitles on YouTube watch pages.

The repository is not a package-manager monorepo. Use the root `Taskfile.yml` as the canonical cross-module command interface. Tool versions are pinned through `mise.toml`.

## Tooling

- Use `mise install` to install the pinned toolchain: Python 3.12, Go 1.22, Node 22, uv, Task, and actionlint.
- Prefer `task <name>` from the repository root for repeatable workflows.
- Use `mise exec -- <command>` when running module commands directly so the pinned toolchain is used.
- Do not commit local secrets from `.env`. Start from `.env.example` when a local environment file is needed.

## Setup Commands

- Install all pinned tools and module dependencies: `task setup`
- Install only toolchains: `task tools`
- Install backend dependencies: `task deps:backend`
- Install whisper dependencies: `task deps:whisper`
- Install extension dependencies: `task deps:extension`

Manual equivalents:

```bash
mise install
cd backend && mise exec -- go mod download
cd whisper && mise exec -- uv sync --dev
cd extension && mise exec -- npm install
```

## Development Workflow

- Start the backend API locally: `task dev:backend`
- Start the extension dev server: `task dev:extension`
- Start the Docker backend: `task docker:up`
- Rebuild and start the Docker backend: `task docker:build`
- Follow Docker backend logs: `task docker:logs`
- Stop Docker backend: `task docker:down`
- Submit a local smoke-test job: `task api:smoke`

`task dev:backend` runs `backend/cmd/server` and prepends `whisper/.venv/bin` to `PATH` so the Go backend can call `whisper-cli`. It uses these defaults unless overridden:

- `LSI_ADDR=127.0.0.1:8080`
- `LSI_DOWNLOAD_TIMEOUT=10m`
- `LSI_WHISPER_MODEL=small`
- `LSI_LLM_BASE_URL=https://api.openai.com`
- `LSI_LLM_TIMEOUT=2m`

For Docker, copy `.env.example` to `.env` and set real values for required settings such as `LSI_LLM_API_KEY`, `LSI_LLM_MODEL`, and `LSI_DOCKER_BIND_HOST`.

The extension popup defaults to `http://127.0.0.1:8080`. The extension manifest only grants host permissions for `localhost` and `127.0.0.1` backend URLs.

## Testing Instructions

- Run all test suites: `task test`
- Run backend Go tests: `task test:backend`
- Run whisper pytest suite: `task test:whisper`
- Run extension Vitest suite: `task test:extension`
- Run extension TypeScript typecheck: `task typecheck`
- Run all tests plus typecheck: `task check`

Direct module commands:

```bash
cd backend && mise exec -- go test ./...
cd whisper && mise exec -- uv run pytest
cd extension && mise exec -- npm run test
cd extension && mise exec -- npm run typecheck
```

Test locations and conventions:

- Go tests live next to implementation files and use `*_test.go`.
- Python tests live under `whisper/tests/` and use `test_*.py`.
- Extension tests live next to source files or entrypoints and use `*.test.ts`.
- Vitest uses `jsdom`, global APIs, and WXT's Vitest plugin from `extension/vitest.config.ts`.
- Add or update tests for behavior changes, especially parser logic, API routes, persistence, job lifecycle, and extension message/storage flows.

To run focused tests, use framework-native filtering from the module directory, for example:

```bash
cd backend && mise exec -- go test ./internal/api -run TestName
cd whisper && mise exec -- uv run pytest tests/test_vtt.py -k cue
cd extension && mise exec -- npx vitest run src/subtitles/vtt.test.ts
```

## Build Commands

- Build all modules: `task build`
- Build backend packages: `task build:backend`
- Build whisper package: `task build:whisper`
- Build Chrome MV3 extension: `task build:extension`

Build outputs:

- Extension production output is generated under `extension/.output/chrome-mv3`.
- Whisper package artifacts are generated by `uv build` under `whisper/dist`.
- Backend image builds use `backend/Dockerfile` with repository root as Docker context.

## Code Style Guidelines

General:

- Make small, surgical changes that directly serve the task.
- Keep module boundaries clear: browser-extension code in `extension/`, API/server code in `backend/`, local transcription CLI code in `whisper/`.
- Prefer existing patterns and naming over introducing new abstractions.
- Do not edit generated or local artifact directories such as `extension/.output/`, `whisper/.venv/`, pytest caches, or compiled binaries.

Backend Go:

- Keep executable entrypoints in `backend/cmd/` and reusable code in `backend/internal/`.
- Use Go's standard formatting: `gofmt` all changed Go files.
- Keep API route behavior covered by tests under `backend/internal/api`.
- Keep store behavior and migrations covered by tests under `backend/internal/store`.

Whisper Python:

- Package source lives under `whisper/src/whisper_cli/`.
- Keep CLI entry behavior in `cli.py`, transcription behavior in `transcribe.py`, and VTT helpers in `vtt.py` unless a change clearly requires otherwise.
- Use `uv` for dependency and command execution.

Extension Vue/TypeScript:

- WXT is configured with `srcDir: 'src'` and `entrypointsDir: '../entrypoints'`; entrypoints live in `extension/entrypoints/`.
- Use `@` imports for `extension/src` where appropriate, matching `wxt.config.ts`.
- Shared UI components live under `extension/src/components/ui/`.
- For extension UI, prefer shadcn-vue components whenever they provide the needed primitive. Only create custom UI components when shadcn-vue has no suitable component.
- Before adding or changing shadcn-vue usage, read the current official documentation at `https://www.shadcn-vue.com/llms.txt`.
- Preserve Chrome MV3 constraints and existing host permissions unless the task explicitly requires manifest changes.

## Environment And Secrets

- `.env.example` documents the expected local Docker/backend environment variables.
- `.env` is local-only and must not be committed.
- `LSI_LLM_API_KEY` is required for real translation flows.
- `HF_TOKEN` is optional and only used to increase Hugging Face Hub download limits.
- Docker persists backend data in the `lsi-data` volume and Hugging Face cache in the `lsi-hf-cache` volume.

## CI/CD

- `.github/workflows/extension-build.yml` runs on `main` pushes affecting `extension/**` or the workflow file, installs dependencies with `npm ci`, runs `npm run build`, and uploads `extension/.output/chrome-mv3` as an artifact.
- `.github/workflows/backend-image.yml` runs on `main` pushes affecting backend, whisper, Docker, or workflow files. It builds and pushes `ghcr.io/<owner>/<repo>-api` using `backend/Dockerfile`.
- Before changing workflow files, validate YAML and action syntax where practical. `actionlint` is available through mise.

## Pull Request Guidelines

- Use `.github/pull_request_template.md` for PR descriptions.
- Include references, summary, file-level notes, validation commands, and review focus areas when relevant.
- Run the narrowest relevant tests during development, then run `task check` before finalizing changes that affect code.
- For Docker/backend/whisper deployment changes, also run the relevant build command such as `task build:backend`, `task build:whisper`, or `task docker:build` when feasible.

## Common Gotchas

- `task dev:backend` expects the whisper virtual environment to exist; run `task deps:whisper` or `task setup` first.
- Extension tests and typechecking call `wxt prepare`, so run them from the module command or via `task` rather than invoking `vue-tsc` without WXT preparation.
- Docker Compose requires `LSI_DOCKER_BIND_HOST` from `.env`; the compose file uses a required-variable expression.
- The extension only supports `localhost` and `127.0.0.1` backend URLs in the current manifest.
- Avoid committing generated artifacts from local builds unless the task explicitly asks for release artifacts.
