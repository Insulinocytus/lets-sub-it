# AGENTS.md

## 适用范围

本文件适用于 `docs/` 下的所有文件。仓库根目录的 `AGENTS.md` 仍然适用；如果两者有冲突，以本文件对 `docs/` 的说明为准。

`docs/` 是面向人类协作者的长期文档区，不是代码实现区。除非任务明确要求，否则不要在编辑文档时顺手修改代码、配置或生成物。

## 文档概览

Lets Sub It 是一个本地优先的 YouTube 字幕生成与翻译工具。`docs/` 记录用户和维护者需要长期阅读的项目知识，按 Diátaxis 四象限组织：

- `tutorials/`：教程，面向第一次加入或从头完成一条学习路径的人。
- `how-to/`：操作指南，面向已经知道目标、需要完成具体任务的人。
- `reference/`：参考资料，记录稳定契约、命令、配置、API、数据模型和目录结构。
- `explanation/`：解释文档，说明架构、边界、处理链路、安全与隐私取舍。
- `superpowers/`：AI 协作过程中生成的规格和计划档案，不是人类长期文档入口。

新增或移动文档前，先确认内容应该进入哪个象限。不要把教程、操作步骤、查表信息和设计解释混在同一篇文档里。

## 语言政策

`docs/` 下的人类长期文档正文、标题、说明、注释应使用中文。

代码、命令、路径、配置键、API 名、专有名词、必须保持原样的引用可以保留英文。

`docs/superpowers/` 是 AI 协作生成的历史档案区。旧档案可保留生成时的语言和格式，除非本次任务明确要求编辑这些档案。

新增或主动编辑的人类文档如果出现非中文 prose，应翻译为自然中文，同时保留技术含义。

## 编辑原则

- 保持文档面向人类阅读。优先写清楚目标、前提、步骤和验证方式，避免代理内部视角。
- 保持改动范围精确。只更新与本次任务相关的文档，不重排无关段落，不顺手统一格式。
- 不要虚构能力、状态或路线图。文档中的 API、配置、命令、支持语言、运行边界和安全说明必须能从代码、README 或现有设计文档中核对。
- 提到安全边界时，保持项目定位准确：后端面向单用户本地自托管，没有公网生产级鉴权保护。
- 提到测试或运行真实链路时，区分离线单元测试和真实下载、模型、LLM 调用。不要让文档暗示单元测试会访问 YouTube、下载模型或调用外部 LLM。
- 链接优先使用相对路径。改名或移动文档时，同步更新入口页和引用方。
- 表格适合稳定查表信息；步骤型内容优先使用短段落和有序列表。

## 常用工作流

从仓库根目录开始查找相关文档：

```bash
rg "关键词" docs README.md backend/README.md whisper/README.md extension/README.md
```

查看 `docs/` 的文档入口和结构：

```bash
find docs -maxdepth 2 -type f -name '*.md' | sort
```

编辑前先确认目标文档类型：

- 学习路径和完整 walkthrough 放在 `docs/tutorials/`。
- 完成具体任务的步骤放在 `docs/how-to/`。
- API、配置、CLI、数据模型、目录结构等稳定事实放在 `docs/reference/`。
- 架构原因、模块边界、处理链路和安全取舍放在 `docs/explanation/`。

## 同步要求

如果文档改动涉及以下内容，检查是否还需要同步 `README.md`、相关模块 README 或其他 `docs/` 页面：

- API 路径、请求、响应、状态流或错误语义。
- 环境变量、默认值、启动命令或 Docker 行为。
- 目录结构、模块职责、runner 边界或扩展通信边界。
- Whisper CLI 参数、输出、退出码或验证规则。
- 支持语言、字幕模式、后端 URL 限制或 manifest 权限。
- 安全与隐私边界。

不要在未核对代码或权威文档的情况下只改一处契约说明。

## 验证说明

文档-only 改动至少运行：

```bash
git diff --check
```

如果改动更新了命令、API 或配置说明，尽量运行对应模块的聚焦验证，或在最终说明中明确未运行的原因：

```bash
cd backend && mise exec -- go test ./...
cd whisper && mise exec -- uv run pytest
cd extension && mise exec -- npm run test
```

如果只是修正文案、链接或结构说明，不需要运行代码测试；但仍要复核 `git diff`，确认没有无关改动或生成物。

## 提交前检查

- 运行 `git status --short`，确认只包含本次任务相关文件。
- 运行 `git diff --check`，修复尾随空白和格式问题。
- 阅读最终 diff，确认每个改动都能追溯到任务目标。
- 最终回复中列出改动文件和实际运行的验证命令。
